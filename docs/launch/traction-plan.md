# 30-day repository traction plan

This plan is designed to test the current growth hypothesis, not promise
virality. The 2026-07-21 baseline is 1 star, 0 forks, 0 subscribers, 22 unique
repository visitors in GitHub's rolling 14-day window, zero direct downloads
across the nine v0.1.3 GitHub release assets, and no comments on the four seeded
Discussions.

## Funnel and targets

Targets are planning goals, not forecasts.

| Stage | Baseline | 30-day test target | Evidence |
|---|---:|---:|---|
| Qualified discovery | 22 unique visitors / 14 days | 200 unique visitors / 14 days | GitHub traffic snapshot |
| First activation | One maintainer clean-install receipt | 10 externally reported clean installs | Issue 11 or a reproducible Discussion reply |
| Proof engagement | 4 unique overview visitors / 14 days | 25 unique overview visitors / 14 days plus 3 shared candidates | GitHub popular content and Show-and-tell posts |
| Participation | 0 external Discussion comments | 3 substantive external comments and 1 external PR | GitHub authorship and review history |
| Physical credibility | No fabricated golden board | No numeric target; preserve blocker honesty until evidence exists | Hash-addressed qualification records |

Do not use raw clone counts as a success metric. They include automation and do
not establish installation, candidate generation, retention, or fabrication.

## Week 0: repair conversion

- Put the current 3D render and interactive review in the first README screen.
- Replace the stale source-only warning with the clean-smoked PyPI quickstart.
- Use the 3D render in the reproducible 20-second demo and artifact receipt.
- Update the description and topics around agentic hardware, electronics, CAD,
  and design automation.
- Configure the 3D render as the repository share image.
- Close the completed interactive-3D issue and open a real newcomer task.

## Week 1: one proof package, several native introductions

- Publish the technical launch post from [`announcement.md`](announcement.md)
  once on the maintainer's primary network.
- Submit a Show HN post with the interactive review as the first link.
- Share a technically adapted version with one KiCad/open-hardware community and
  one agent-tooling community after checking each community's current rules.
- Answer every substantive question with a link to the exact gate, artifact, or
  limitation. Do not paper over the missing physical evidence.

## Week 2: demonstrate breadth

- Publish one second supported-family walkthrough, chosen for a visibly
  different use case from the RP2040 USB board.
- Show the prompt, generated artifacts, 3D result when available, and remaining
  blocker report in the same compact format.
- Link the walkthrough back to the PyPI quickstart and Show-and-tell category.

## Week 3: turn attention into contribution

- Review and respond to issue 11 install receipts within one working day.
- Convert any repeated install friction into a focused documentation or
  packaging issue with an executable acceptance test.
- Highlight one contributor result in Discussions and the next artifact-led
  update, with permission and attribution.

## Week 4: decide from evidence

Capture the same GitHub traffic, release-download, issue, Discussion, and PR
metrics. Compare the funnel stages, not only the star count:

- High exposure and low activation means the first-use path or promise needs
  another test.
- Successful activation but no shared candidates means the generated result or
  sharing loop needs work.
- Shared candidates but no contribution means newcomer scope and maintainer
  response time need work.
- Low exposure after the launch package means distribution, audience selection,
  or the headline hypothesis needs a new experiment.

Record results with dates and preserve failed experiments. Fame is an outcome;
the controllable system is proof quality, activation speed, distribution fit,
and community response.
